# Home Screen Spec Split Session — 2026-05-07

## Why this session happened

Home-screen behavior was still spread across multiple feature specs even after the cross-cutting ADRs were accepted. That made it too easy for the dashboard rows and badge behavior to drift apart as the specs evolved.

The goal of this session was not to invent a new feature. It was to give the existing home behavior one canonical home.

## Decisions made

### Home gets its own feature spec

- Added a dedicated `specs/features/home.md`
- Home is now the canonical source for:
  - `Recently Added`
  - `Continue Watching`
  - the service health badge

### Browse & Play no longer owns home rows

- `browse-play.md` now describes library browsing, item details, and playback handoff
- Home entry points must route into the same detail and play flows defined there
- `library.resume` is treated as a Home concern rather than a Browse & Play-owned row requirement

### Health no longer owns badge behavior

- `health.md` now owns the detailed health view and the last-known health state behind it
- `home.md` owns when and how the badge is surfaced to users

### Link-gated personalized home behavior

- Home rows that depend on user-specific upstream context must fail explicitly when the required external account link is missing
- No guest, shared-account, or anonymous fallback was introduced

### Health spec cleanup

- The health spec now matches the accepted realtime and caching ADRs
- The previous "configurable interval" wording was removed in favor of the shared `60s` default
- Resolved badge-visibility questions were removed from the open-questions section

## Immediate follow-up chosen from this session

- Use `home.md` as the reference point when finishing the remaining feature-spec cleanup
- Run a broader spec review pass next to identify which remaining feature specs are still blocked only by feature-level product questions
