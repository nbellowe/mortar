# Docker Image and CI Pipeline — 2026-05-08

## What was built

- **Dockerfile** — 3-stage multi-stage build: Node 20 Alpine builds the Expo web export, Golang 1.22 Alpine (with CGO for go-sqlite3) compiles the server binary with the web export baked in via `go:embed`, and the final Alpine runtime image contains only the binary.
- **CI workflow** (`.github/workflows/ci.yml`) — `go test ./...` and `tsc --noEmit` run on every PR; on merge to `main` the Docker image is built and pushed to `ghcr.io/nbellowe/mortar` tagged `latest` and the short commit SHA.
- **SPA fallback handler** — `router.go` now serves static files from the embedded FS and falls back to `index.html` for any path that doesn't match a real file, enabling Expo Router's client-side navigation.
- **`--db` flag** — configurable SQLite path (defaults to `mortar.db`; Docker deployment uses `/data/mortar.db`).
- **`web/index.html` placeholder** — checked in so `go:embed web` compiles in dev without a prior Expo build. Replaced by the real Expo output during `docker build`.

## Key decisions

**Single image, not backend + frontend split.** The Go binary embeds the entire Expo web export via `go:embed`. One image, one port, no nginx sidecar. Appropriate for homelab scale; avoids extra operational surface.

**GHCR over Docker Hub.** Uses the built-in `GITHUB_TOKEN` — no external registry credentials needed in CI. Image is `ghcr.io/nbellowe/mortar`.

**`/data/` mount convention.** Both the config file and the SQLite database live under `/data/`. Users bind-mount their config as `/data/config.yaml:ro` and use a named volume for the DB. This gives one volume mount point, keeping the `docker run` invocation short.

**typescript `~5.8` instead of `~6.0`.** Expo 55's internal packages peer-require `^5.0.0`, which excludes TypeScript 6.x. `npm ci` in the Docker build enforces strict lock file integrity, so the version mismatch surfaced immediately in testing. Pinned to 5.8 to satisfy the constraint; can be revisited when Expo drops the constraint.

**`npm ci --prefer-offline`** in the Dockerfile. Avoids unnecessary network round-trips on re-builds when the layer is cached. Falls back to network for cold builds.

## What was ruled out

**`CGO_ENABLED=0`** — go-sqlite3 requires CGO. The Go builder stage installs `gcc` and `musl-dev` and builds with `CGO_ENABLED=1`. Runtime stage is plain Alpine (musl libc is included).

**Separate `/config/` mount.** The original architecture spec example showed `-v ./mortar.yaml:/config/mortar.yaml`. Consolidated to `/data/` to simplify deployment; spec updated to match.
